### 用例 1：并发压力测试
基础信息
1. 测试对象：OpenClaw 任务
2. 任务说明：以纯后台模式执行，无完整浏览器进程创建，单进程内存开销较低
3. 单轮时长：10s，超时统一回收进程
4. 监控方式：通过 mem_monitor.sh 脚本记录内存变化，top 命令实时观测系统状态

用例列表
|用例 ID	|并发数量	|预期结果	|实际结果	|测试状态|
|-------|-----|---------------------------------|---------------------|----|
|TC-001	|100	|进程正常启动，压测完整执行，无异常	|进程正常拉起，服务稳定运行；内存占用稳定，无异常波动	|通过|
|TC-002	|200	|进程正常启动，系统负载平稳，执行成功	|进程正常拉起，服务稳定运行；内存随并发线性增长，压测结束后正常回落	|通过|
|TC-003	|300	|进程正常运行，系统临近资源阈值，整体稳定	|进程正常拉起，服务稳定运行；内存占用接近临界值，压测结束后可正常回收	|通过|
|TC-004	|350	|任务可正常执行	|系统资源压力明显上升，部分任务执行效率下降；内存持续高位，压测结束后回收缓慢，存在触发 OOM 机制的风险	|不通过|

观测指标
1. 进程启停状态：是否出现异常退出、闪退
2. 系统资源占用：内存随并发的变化趋势、压测前后是否正常回落，有无持续增长或无法回收迹象
3. 压测流程完整性：整轮压测能否按设计时长正常执行
4. 系统并发承载临界值：定位系统从稳定运行到性能下降的拐点


### 用例 2：心跳风暴压测
用例 ID：TC-DAY12-001
测试场景：150 并发心跳风暴

前置条件：
1. 系统空闲内存 ≥ 1.5GB，CPU 空闲率 ≥ 90%
2. stress_test_suite.sh、mem_monitor.sh 脚本已准备就绪
3. 日志文件 /var/log/openclaw.log 已清空

执行步骤：
1. 终端 1 启动内存监控：./mem_monitor.sh
2. 终端 2 启动 CPU 监控：top
3. 终端 3 启动日志监控：tail -f /var/log/openclaw.log | grep -E "ERROR|WARN|retry|heartbeat"
4. 终端 4 执行压测：./stress_test_suite.sh 150 heartbeat                    //scripts里的stress_test_suite.sh
5. 等待压测执行完成，停止所有监控进程；记录压测后内存、CPU 基线，检查日志内容

预期结果：
1. 压测期间内存正常上涨，结束后回落，无泄漏
2. 无 pthread_create 或 OOM 错误
3. 日志无报错、无警告、无重试风暴 / 日志风暴
4. 服务无崩溃、无重启

实际结果：与预期结果一致，测试通过。
